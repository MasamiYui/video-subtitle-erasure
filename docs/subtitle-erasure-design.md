# 视频字幕擦除方案设计

## 1. 文档目的

本文档用于明确 `video-subtitle-erasure` 的下一阶段实现方案，重点解决当前版本中“字幕仍有残留”的问题，并同时定义一个可用的 Web 前端交互方案，供实现前确认。

本文档关注的是：

- 识别为什么现有方案会漏擦
- 确定下一版后端处理链路
- 明确自动、半自动、手动三种工作模式
- 设计一个适合调参与预览的 Web 页面
- 规划实现阶段和验收标准

本文档暂不讨论：

- 云端部署
- 多用户协作
- 超出 `mp4` 的输入格式
- 最终商业化 UI 细节

## 2. 目标与非目标

### 2.1 目标

- 输入一个本地 `mp4` 视频，输出擦除原始硬字幕后的视频
- 默认支持自动检测字幕位置，无需依赖 `.srt`
- 对固定底部字幕的视频，提供可控的半自动/手动增强模式
- 尽可能减少以下问题：
  - 字幕残留
  - 擦除区域闪烁
  - 误伤前景人物或场景细节
- 提供可视化前端，便于用户先预览、再全片处理

### 2.2 非目标

- 不追求一次性解决所有复杂场景
- 不要求第一版就接入最重的视频补全大模型
- 不要求自动适配所有样式的台标、水印、弹幕

## 3. 当前问题分析

当前基线版本已经具备自动定位和擦除能力，但仍会出现残字。根因至少分为三类。

### 3.1 时间漏检

当前字幕的时间段主要来自抽样检测再合并，容易出现：

- 字幕实际开始更早，但未被采样命中
- 字幕实际结束更晚，但结束边界收得过快
- 相邻两句字幕切换时，中间短空档导致被切成多个片段，覆盖不足

表现：

- 一句字幕刚出现的前几帧没擦
- 一句字幕消失前的后几帧没擦

### 3.2 空间漏检

当前每帧使用的是“检测框 + 简单细化 mask”的思路，容易漏掉：

- 白字的抗锯齿边缘
- 黑色描边
- 阴影
- 半透明边缘
- 紧挨在一起的字之间的空隙

表现：

- 同一帧里还有零散字块没擦掉
- 擦掉主体，但描边还留着

### 3.3 擦除后端的能力边界

当前基线采用的是局部 OpenCV inpaint，优点是稳定、简单、快；缺点是：

- 花纹背景下容易出现糊带
- 前景人物压住字幕带时，修复质量有限
- 无法充分利用视频时域信息

结论：

- 当前最急需解决的，不是“立刻换更强补全模型”
- 而是先把“字幕时间召回率”和“字幕像素召回率”做满

## 4. 方案原则

下一版设计遵循以下原则：

1. 先把字幕 mask 做准，再谈更强 inpaint
2. 不做“全片逐帧 OCR 认字”，而是做“候选片段逐帧检测/分割”
3. 自动模式要能跑，半自动模式要更稳，手动模式要兜底
4. 擦除后必须有复检闭环
5. Web 页面必须支持局部预跑，避免全片跑完才发现效果不好

## 5. 总体方案

建议把系统分为五层：

1. ROI 层：确定字幕可能出现的区域
2. 时间层：确定哪些帧需要处理
3. 空间层：为每一帧生成尽可能完整的字幕 mask
4. 擦除层：根据 mask 执行修复
5. 复检层：检查是否有残字，必要时二次处理

### 5.1 总体处理链路

```text
输入视频
  -> 读取视频元信息
  -> 确定字幕 ROI（自动 / 半自动 / 手动）
  -> 粗扫候选字幕时间段
  -> 候选时间段内逐帧检测 / 分割
  -> 生成逐帧字幕 mask
  -> 扩张时间边界与空间边界
  -> 按片段执行擦除
  -> 残字复检
  -> 对失败片段执行二次重跑
  -> 输出视频与调试信息
```

## 6. 工作模式设计

### 6.1 自动模式 `auto`

适用场景：

- 用户不想人工标注
- 字幕大致固定在底部、中部或顶部某条区域

流程：

- 自动发现字幕 ROI
- 粗扫时间段
- 候选区间内逐帧处理
- 残字复检

优点：

- 用户成本最低

缺点：

- 对复杂场景不一定最稳

### 6.2 半自动模式 `semi-auto`

适用场景：

- 用户知道字幕长期位于某个区域
- 希望减少误检和漏检

流程：

- 用户手动框选 1 到 3 个字幕区域
- 系统仅在选区内做时间检测和逐帧 mask 生成
- 擦除后复检

优点：

- 极大降低搜索空间
- 更适合固定底部字幕视频

缺点：

- 需要一次人工参与

### 6.3 手动固定模式 `manual-fixed`

适用场景：

- 字幕区域完全固定
- 用户只想把某块区域中的字幕无脑擦掉

流程：

- 用户手动框选固定带状区域
- 系统对该区域内每个候选帧强制处理

优点：

- 最稳的兜底模式

缺点：

- 误伤风险最高
- 需要更强的结果预览和局部测试

## 7. 后端技术方案

### 7.1 ROI 层

建议支持两种来源：

- 自动 ROI：复用 `subtitle-ocr` 当前的 anchor 发现逻辑
- 手动 ROI：用户在预览界面画框

ROI 数据结构建议：

```json
{
  "mode": "auto | semi-auto | manual-fixed",
  "regions": [
    { "x1": 0.15, "y1": 0.78, "x2": 0.85, "y2": 0.95 }
  ]
}
```

其中坐标建议用相对比例，避免分辨率切换问题。

### 7.2 时间层

#### 7.2.1 不建议全片逐帧 OCR

原因：

- 成本太高
- OCR 关注“识别文字”，而不是“找字幕像素”
- 很多问题本质是 mask 不完整，不是文本内容不知道

#### 7.2.2 建议采用“粗到细”

流程：

- 粗扫阶段
  - 每 `N` 帧做一次轻量文本性检测
  - 目的是找出“疑似有字幕”的候选区间
- 细扫阶段
  - 只在候选区间内逐帧处理

粗扫阶段可用信号：

- OCR 文本检测
- 边缘密度
- 亮字分布
- 文本形态先验

#### 7.2.3 建议引入滞后关闭

问题：

- 现在的时间边界太容易提前结束

改进：

- 一旦进入“字幕存在”状态，不要因为 1 帧没检测到就结束
- 只有连续 `K` 帧都确认无字幕，才关闭该片段

建议初始参数：

- `open_threshold`: 检测到 1 帧即可打开
- `close_threshold`: 连续 3 到 5 帧无字幕才关闭
- `time_expand_pre`: 向前补 2 到 4 帧
- `time_expand_post`: 向后补 2 到 4 帧

这一步主要解决“实际字幕展示时间高于检测时间”的问题。

### 7.3 空间层

核心原则：

- 逐帧生成 mask
- 目标是找到字幕像素，而不是识别出字幕文本

#### 7.3.1 候选来源

建议组合以下信号：

- OCR / 文本检测框
- 多边形检测结果
- 亮字阈值
- 黑边与阴影的形态学扩张
- 连通域合并
- 相邻帧 mask 传播

#### 7.3.2 mask 构建策略

建议处理顺序：

1. 用检测框或多边形限制范围
2. 在范围内提取亮字核心区域
3. 对黑边和阴影做单独扩张
4. 对分离字块做桥接和闭运算
5. 与相邻帧 mask 做局部并集

#### 7.3.3 两级 mask

建议把每帧字幕 mask 分成两层：

- `core mask`：高置信文字主体
- `soft mask`：描边、阴影、半透明边缘

后端擦除时：

- 对 `core mask` 必擦
- 对 `soft mask` 结合片段上下文和后端能力决定是否一起擦

### 7.4 擦除层

建议分两档。

#### 7.4.1 快速模式

后端：

- OpenCV / LaMa 类

适用：

- 固定字幕
- 纯底部字幕
- 快速预跑

优点：

- 快
- 稳
- 易于工程化

#### 7.4.2 高质量模式

后端候选：

- STTN
- ProPainter
- Blind / video decaptioning 模型

适用：

- 花纹背景
- 运动背景
- 人物压住字幕带

工程建议：

- 不全片调用
- 只对“字幕片段”调用
- 只在自动复检失败时升级使用

### 7.5 复检层

这是下一版必须有的能力。

流程：

1. 第一遍擦完后，对输出视频在 ROI 内做一次轻量检测
2. 若检测到残字：
   - 标记对应时间段
   - 自动加大 mask
   - 提高逐帧密度
   - 必要时切换更强后端
3. 仅重跑失败片段

这一步决定系统能否从“可跑 demo”走向“可用工具”。

## 8. Web 页面方案

页面风格参考：

- [subtitle-ocr/static/index.html](/Users/masamiyui/OpenSoureProjects/Forks/subtitle-ocr/static/index.html:1)

建议保留其优点：

- 单页应用
- 白底卡片式布局
- 参数区和结果区清晰
- 进度条与状态明确
- 视频预览支持叠加可视元素

### 8.1 页面结构

建议单页分为五块：

1. 顶部状态栏
2. 上传与模式选择区
3. 参数区
4. 视频预览与标注区
5. 结果与导出区

### 8.2 页面模块

#### 8.2.1 上传区

支持：

- 本地 `mp4` 上传
- 拖拽上传
- 显示视频时长、尺寸、帧率

#### 8.2.2 模式区

建议提供：

- 自动模式
- 半自动模式
- 手动固定模式

并配一个后端质量开关：

- 快速
- 高质量

#### 8.2.3 参数区

基础参数建议：

- 字幕位置先验：`auto/bottom/middle/top`
- 粗扫采样间隔
- 关闭滞后帧数
- 时间边界扩张帧数
- mask 横向扩张
- mask 纵向扩张
- 擦除模式：快速 / 高质量

高级参数建议折叠显示：

- ROI 搜索比例
- 亮字阈值
- 描边扩张强度
- soft mask 强度
- 残字复检阈值
- 二次重跑策略

#### 8.2.4 预览区

预览区是这个项目的关键。

建议同时支持：

- 原视频预览
- 当前帧 ROI 叠加
- 当前帧检测框 / 多边形叠加
- 当前帧 mask 叠加
- 擦除结果预览
- 原图 / 结果对比滑杆

#### 8.2.5 标注交互

在半自动和手动模式下，允许：

- 在视频画面上框选矩形区域
- 支持添加多个框
- 删除框
- 调整框大小

建议还支持：

- 选取一个时间点，执行“局部预跑 3 到 8 秒”
- 用户满意后再跑全片

#### 8.2.6 结果区

显示：

- 当前任务状态
- 处理进度
- 当前阶段
- 输出视频下载按钮
- 调试包下载按钮

调试包建议包含：

- ROI
- 检测到的时间段
- 每段的关键帧截图
- mask 叠加图
- 复检结果

## 9. 后端接口草案

建议采用和 `subtitle-ocr` 相近的 FastAPI 风格。

### 9.1 任务接口

- `POST /api/v1/erase/extract`
  - 上传视频并同步处理短片段
- `POST /api/v1/erase/extract/async`
  - 上传视频并异步全片处理
- `GET /api/v1/erase/status/{task_id}`
  - 查询进度
- `GET /api/v1/erase/download/{task_id}`
  - 下载结果视频
- `GET /api/v1/erase/debug/{task_id}`
  - 下载调试信息

### 9.2 预览接口

- `POST /api/v1/erase/preview`
  - 对用户指定时间段进行局部预跑

请求体建议包含：

- 视频路径或上传文件
- 模式
- ROI
- 时间范围
- 基础参数

### 9.3 ROI 接口

- `POST /api/v1/erase/roi/suggest`
  - 自动建议字幕区域

### 9.4 复检接口

- `POST /api/v1/erase/recheck`
  - 对指定片段执行残字复检

## 10. 数据与调试产物

建议保留以下中间信息，方便调参与问题定位：

- 视频元信息
- 选中的 ROI
- 候选字幕时间段
- 每帧检测结果
- 每帧最终 mask
- 擦除前后关键帧
- 残字复检命中结果

建议输出目录：

```text
output/
  task-xxx/
    result.mp4
    debug/
      metadata.json
      roi.json
      timeline.json
      masks/
      keyframes/
      residual_check.json
```

## 12. 分阶段实现建议

### 阶段 1：工程可用版

目标：

- 自动 / 半自动 / 手动三种模式
- 候选区间逐帧检测
- 时间边界扩张与滞后关闭
- 更完整的逐帧 mask
- 基础 Web 页面

不做：

- 重型视频补全模型
- 自动二次重跑

### 阶段 2：可靠性增强版

目标：

- 残字复检
- 对失败片段自动二次重跑
- 调试包下载
- 局部预跑能力完善

### 阶段 3：高质量版

目标：

- 接入更强的视频补全后端
- 只对失败片段或指定片段启用高质量模式

## 13. 验收标准

建议按三档场景验收：

### 13.1 简单场景

- 固定底部字幕
- 背景平滑
- 无人物压住字幕

标准：

- 基本无残字
- 几乎无明显糊带

### 13.2 中等场景

- 固定底部字幕
- 背景有轻微纹理或运动

标准：

- 允许极少量局部瑕疵
- 不应存在明显残字

### 13.3 困难场景

- 人物或前景压住字幕带
- 复杂纹理背景
- 快速切换字幕

标准：

- 允许轻微修补痕迹
- 不能保留完整可读文字

## 14. 待确认项

以下问题建议在实现前确认：

1. 第一版 Web 页面是否只做本地单用户使用
2. 第一版是否接受“快速模式”和“高质量模式”只提供一个默认后端
3. 手动画框是否只支持矩形，不支持多边形
4. 第一版残字复检是否只做检测，不自动二次重跑
5. 第一版是否只优先服务固定底部字幕视频

## 15. 建议结论

建议按以下方向推进：

- 保留自动模式，但增加半自动和手动固定模式
- 不走“全片逐帧 OCR 认字”，改走“候选区间逐帧检测/分割”
- 先把时间边界和 mask 做准
- Web 页面优先支持 ROI 框选、局部预跑和结果对比
- 重型视频补全后端放到后续版本，只对疑难片段启用

如果这份方案确认无误，下一步可以基于本文档拆成：

- 后端模块设计
- API 文档
- Web 页面原型与实现清单
